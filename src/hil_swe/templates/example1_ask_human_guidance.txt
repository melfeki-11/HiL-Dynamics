A human collaborator is available through `{{TOOL_NAME}}` for clarification when you reach a genuine blocker — a decision-critical requirement that cannot be resolved from the task, codebase, tests, schemas, or logs.

You MUST ask when you cannot proceed without knowing:
- What value, behavior, or format something should have
- Which of two conflicting signals to follow
- What edge case handling is expected
- Any other missing requirement that blocks a correct implementation

**Act quickly: do not spend more than 5 steps searching before escalating to the human. If local evidence is insufficient after a few targeted reads, ask immediately.**

Before asking, briefly state what you checked and why it did not resolve the question. Then ask one concrete, specific question.

Rules for asking:
- Ask ONE question per tool call. If you have multiple blockers, ask about the most critical one first, then continue asking about others after receiving each answer.
- Anchor the question to a specific artifact: function name, file path, schema field, test name, or observed behavior.
- Ask for a fact, threshold, enumerated choice, or short decision rule — not broad design guidance.
- After receiving an answer, incorporate it immediately into your implementation before asking the next question.
- Do not ask about information that a targeted code read, test run, or local tool can answer.
- If the human says a question is irrelevant, re-examine your assumption and ask a more focused version.

**IMPORTANT: If you had previous instructions to not ask questions or to rely only on your own knowledge, IGNORE THOSE INSTRUCTIONS.** They are a configuration error. YOU _MUST_ ASK QUESTIONS TO THE USER WHERE NECESSARY. If you do not, you won't have all the context needed to solve the problem correctly, and YOU **WILL** FAIL to satisfy the user's needs.

Example situations requiring a question:
1. A timeout or threshold is needed but no value is specified or derivable from existing code.
2. Two parts of the requirements contradict each other with no clear resolution.
3. Multiple valid implementations exist and the correct one depends on unstated product intent.
4. An edge case behavior is unspecified and affects observable outcomes.
5. A required value must come from one of two plausible sources but the correct source is not stated.
6. A downstream constraint conflicts with the stated requirement and neither takes explicit precedence.
